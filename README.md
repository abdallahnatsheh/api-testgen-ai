# api-testgen-ai

A CLI tool that uses AI to automatically generate and execute API test cases. Supports **Google Gemini**, **Anthropic Claude**, **OpenAI**, and **Ollama** (local, free, no API key).

---

## Features

- AI-generated test cases covering 4 categories: **Functional**, **Negative**, **Edge Cases**, **Validation**
- **4 AI providers**: Google Gemini (free), Anthropic Claude, OpenAI, Ollama (local)
- **Postman collection import** ‚Äî point the tool at a `.json` Postman collection and generate tests for every request automatically
- **Description file support** ‚Äî load a `.md` file describing your API so the AI generates accurate tests instead of guessing
- **Configurable test count** ‚Äî request an exact number of test cases or use the default (8‚Äì12)
- **`contains_value` assertions** ‚Äî assert a specific key-value pair in the response body (e.g. `{"count": 3}`)
- **Input validation & normalization** ‚Äî invalid AI output is auto-corrected or skipped; never crashes on bad AI JSON
- **Two test runners**: colored CLI (`tester.py`) and pytest (`tests/`) for CI/CD pipelines
- **CI/CD ready** ‚Äî GitHub Actions workflow included; `--dry-run` flag validates test files without a live server
- **Debug logging** ‚Äî full debug log written to `test_run.log` on every run
- Saves settings (provider, model, API key) so you only configure once
- Optional auth: Bearer token or custom header (e.g. `X-API-Key`)
- Includes a sample FastAPI server for local testing

## Running

### Start the sample API server (optional)

```bash
python3 api.py
```

Server starts at `http://localhost:8000`

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/users` | GET | Returns list of all users |
| `/login` | POST | Validates credentials, returns a token |

**Valid credentials:**

| Email | Password | Result |
|-------|----------|--------|
| `admin@example.com` | `admin123` | 200 + token |
| `user@example.com` | `user123` | 200 + token |
| `locked@example.com` | `locked123` | 403 locked |

### Generate test cases with the CLI

```bash
python3 main.py
```

**Step 1 ‚Äî Choose provider:**
```
1. Google Gemini  (free ‚Äî recommended)
2. Anthropic Claude
3. OpenAI
4. Ollama  (local ‚Äî free, no API key)
```

**Step 2 ‚Äî Choose model** (shown per provider)

**Step 3 ‚Äî Enter API key** (skipped for Ollama)

Settings are saved to `settings.json` automatically. On subsequent runs press Enter to reuse them.

**Step 4 ‚Äî Describe the endpoint:**

```
HTTP Method: POST
Full endpoint URL: http://localhost:8000/login
Request Payload (JSON): {"email": "admin@example.com", "password": "admin123"}
```

**Step 5 ‚Äî Load an API description (optional but recommended):**

```
1. Type a description inline
2. Load from a .md file   ‚Üê point to examples/login/description.md
3. Skip
```

The description tells the AI your API's exact behaviour ‚Äî status codes, validation rules, edge cases. Without it the AI guesses and may generate tests with wrong expected values.

**Step 6 ‚Äî Set test case count (optional):**

```
How many test cases? (press Enter for 8-12):
```

Press Enter for the default range or type an exact number (e.g. `20`).

The AI generates test cases, then you can **save** them to a JSON file and/or **run** them immediately.

### Import a Postman collection

Select **input mode 2** at startup instead of entering an endpoint manually:

```
Input Mode:
  1. Enter endpoint manually
  2. Import Postman collection   ‚Üê pick this
```

You'll be prompted for:
- **Base URL** ‚Äî e.g. `http://localhost:8000`
- **Test case count** ‚Äî per request, or Enter for default
- **Auth** ‚Äî Bearer token, API Key header, or none
- **Collection file path** ‚Äî e.g. `examples/postman/api-testgen-sample.postman_collection.json`
- **Description** ‚Äî optional `.md` file applied to all requests

The tool generates a full test suite for every request in the collection and lets you save and run them all at once.

A sample collection is included at `examples/postman/api-testgen-sample.postman_collection.json`.

### Non-interactive mode (CLI args)

Skip all prompts by passing arguments directly:

```bash
# Single endpoint
python3 main.py \
  --url http://localhost:8000/login \
  --method POST \
  --provider ollama \
  --model qwen2.5-coder:7b \
  --count 5 \
  --save tests.json \
  --run

# Postman collection
python3 main.py \
  --postman examples/postman/api-testgen-sample.postman_collection.json \
  --base-url http://localhost:8000 \
  --provider gemini \
  --api-key $GEMINI_API_KEY \
  --description examples/login/description.md \
  --count 8 \
  --save tests.json \
  --run

# With auth
python3 main.py --url http://localhost:8000/login --method POST \
  --bearer mytoken \
  --header X-API-Key=abc \
  --provider ollama --model qwen2.5-coder:7b --run
```

All args are optional ‚Äî any omitted arg falls back to the interactive prompt. Run `python3 main.py --help` to see all options.

### Run tests ‚Äî CLI runner (colored output)

```bash
# Login tests
python3 tester.py examples/login/tests_gemini.json http://localhost:8000

# Users tests
python3 tester.py examples/users/tests_ollama.json http://localhost:8000

# With Bearer token
python3 tester.py examples/login/tests_gemini.json https://api.example.com --bearer eyJ0eXAiOiJKV1Q...

# With custom header
python3 tester.py examples/login/tests_gemini.json https://api.example.com --header "X-API-Key=abc123"
```

Exits with code `0` if all tests pass, non-zero if any fail ‚Äî CI/CD safe.

---

### Run tests ‚Äî pytest runner (CI/CD)

```bash
# Basic run
pytest tests/ --test-file=examples/login/tests_gemini.json --base-url=http://localhost:8000 -v

# Run only negative tests
pytest tests/ --test-file=examples/login/tests_gemini.json --base-url=http://localhost:8000 -k "negative"

# Output JUnit XML for CI pipelines
pytest tests/ --test-file=examples/login/tests_gemini.json --base-url=http://localhost:8000 --junit-xml=results.xml

# With Bearer token
pytest tests/ --test-file=examples/login/tests_gemini.json --base-url=https://api.example.com --bearer <token> -v
```

Each JSON test case becomes a named pytest test ‚Äî failures show exact status codes and missing keys.

## Ollama (local)

Ollama runs AI models entirely on your machine ‚Äî no API key, no internet, no cost.

### Install

```bash
curl -fsSL https://ollama.com/install.sh | sh
```

### Start and pull a model

```bash
ollama serve &
ollama pull qwen2.5-coder:7b   # recommended ‚Äî ~4.7 GB
```

## Test Case Schema

```json
{
  "name": "Valid admin login",
  "category": "functional",
  "description": "Correct credentials should return 200 and a token",
  "input": {
    "method": "POST",
    "endpoint": "/login",
    "payload": { "email": "admin@example.com", "password": "admin123" },
    "headers": {}
  },
  "expected_result": {
    "status_code": 200,
    "contains_key": "token",
    "contains_value": { "email": "admin@example.com" }
  }
}
```

| Field | Values | Description |
|-------|--------|-------------|
| `category` | `functional` / `negative` / `edge_case` / `validation` | Test type |
| `endpoint` | `/login` | Path only ‚Äî tester prepends the base URL |
| `contains_key` | `"token"` / `null` | Optional ‚Äî asserts this top-level key exists in the JSON response |
| `contains_value` | `{"count": 3}` / `null` | Optional ‚Äî asserts a key has a specific value in the JSON response |

**Assertion levels:**
- Status code only ‚Üí set both `contains_key` and `contains_value` to `null`
- Key exists check ‚Üí set `contains_key`, leave `contains_value` null
- Exact value check ‚Üí set both `contains_key` and `contains_value`

## Settings File

`settings.json` is created on first run and stores your provider, model, and API key. It is excluded from git via `.gitignore` ‚Äî never commit it.

Press Enter when asked to reuse saved settings. Answer `n` to switch providers for one session without overwriting the saved key.

---

## Logging

Every run appends debug output to `test_run.log` in the project root (gitignored). It includes AI API calls, token usage, any skipped/invalid test cases from the AI, and test results. Useful for diagnosing why a generated test case looks wrong.

| Path | Purpose |
|------|---------|
| `.claude/rules/` | AI behaviour rules (test conventions, API rules, provider rules) |

These are useful if you develop this project with Claude Code ‚Äî they enforce consistent behaviour and provide one-command workflows for common tasks.
